# 설계 스펙 — 서울 핫플 100 (토스 앱인토스 미니앱)

- 날짜: 2026-07-09
- 상태: 승인 대기
- 상세 문서: [PRD](../../PRD.md) · [아키텍처](../../architecture.md) · [디자인 가이드](../../design-guide.md) · [로드맵](../../roadmap.md)

## 무엇을 만드는가

서울시 실시간 도시데이터(citydata_ppltn, 121개 핫스팟)를 추정 인구 기준으로 정렬해 "지금 서울에서 사람이 가장 많은 곳 TOP 100"을 보여주는 토스 미니앱. 배너 광고로 수익화한다.

## 브레인스토밍에서 확정된 결정

| 질문 | 결정 | 기각한 대안 |
|---|---|---|
| 랭킹 기준 | 실시간 추정 인구(AREA_PPLTN_MAX) 순위 | 급상승 순위(히스토리 저장 필요), 혼잡도 혼합(동점 과다), 한산한 곳(컨셉 역행) |
| MVP 범위 | 랭킹 + 상세 2화면 | 랭킹만(재방문 약함), 즐겨찾기(v2), 게임 요소(무거움) |
| 데이터 수집 | GitHub Actions cron 정적 생성 → GitHub Pages | 집계 서버(운영비), 클라이언트 직접 수집(세션당 121콜, 심사 리스크) |
| 플랫폼 | Granite React Native + TDS | 웹뷰(네이티브 체감 열세) |
| 1차 목표 | 수익화 | — |
| 수익 모델 | 배너 광고(InlineAd) 3지면, v1 | 보상형+프리미엄(범위 초과→v2), IAP(파트너 서버+mTLS 필요→기각) |

## 아키텍처 요약

```
GitHub Actions (30분 cron, 01~05 KST 제외)
  → 서울시 API 121곳 수집 (부분 실패 허용, 직전 값 유지)
  → 정렬 + 직전 스냅샷 diff(순위 변동) → ranking.json (zod 검증)
  → GitHub Pages 배포 (ACAO: *, 앱인토스 origin 요건 충족)

Granite RN 앱
  → ranking.json 1회 fetch (useRanking 훅: ok/error/stale/night)
  → HomeScreen (HOT 100 + 카테고리 필터 + 광고 2지면)
  → DetailScreen (혼잡도 상세 + 연령/성별 분포 + 12시간 예측 + 광고 1지면)
```

런타임 서버 0대. 상세 데이터는 리스트 JSON에 임베드되어 추가 요청 없음.

## 에러 처리

- 파이프라인: 장소별 재시도 2회 → 실패 시 직전 값 + stale 마킹 → 3회 연속 실패 시 랭킹 제외. 대량 실패 시 산출물 미갱신(이전 JSON 서비스 유지)
- 앱: fetch 실패(재시도 버튼) / 30분 지연(배지) / 새벽 01~05시(안내) / 광고 실패·구버전(영역 접기)

## 테스트

- pipeline: vitest 단위+통합 (정렬, diff, 실패 처리, 스키마) — TDD
- app: jest 로직 계층 (useRanking, 유틸) — TDD, 커버리지 80%+
- 시각·광고·실기기: 샌드박스 앱 수동 확인 (자동화 불가 영역)
- 계약: ranking.json zod 스키마를 pipeline/app 이 공유

## 확정 사항 (이전 열린 이슈)

1. 서비스명: **서울 핫플 100** (콘솔 appName `seoul-hot-100`)
2. 갱신 주기: **30분** (자체 키 약 4,840콜/일, Phase 1에서 할당량 여유 확인)
3. 수집 경로: **자체 서울시 API 키** (GitHub Secrets, 프록시 미사용)
